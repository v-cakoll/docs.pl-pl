<span data-ttu-id="4274a-173">Ponadto w przypadku transakcji zatwierdzania w stylu KWASowym lub dwufazowym nie są one bezpośrednio związane z zasadami mikrousług. Większość baz danych NoSQL (takich jak Azure Cosmos DB, MongoDB itp.) nie obsługuje transakcji zatwierdzania dwuetapowego, typowych w scenariuszach dystrybuowanych baz danych.</span><span class="sxs-lookup"><span data-stu-id="4274a-173">Moreover, ACID-style or two-phase commit transactions are not just against microservices principles; most NoSQL databases (like Azure Cosmos DB, MongoDB, etc.) do not support two-phase commit transactions, typical in distributed databases scenarios.</span></span> <span data-ttu-id="4274a-174">Jednak zachowanie spójności danych w ramach usług i baz danych jest niezbędne.</span><span class="sxs-lookup"><span data-stu-id="4274a-174">However, maintaining data consistency across services and databases is essential.</span></span> <span data-ttu-id="4274a-175">To wyzwanie dotyczy również kwestii propagowania zmian w wielu mikrousługach, gdy pewne dane muszą być nadmiarowe, na przykład gdy trzeba mieć nazwę lub opis produktu w mikrousłudze wykazu i koszyku mikrousług.</span><span class="sxs-lookup"><span data-stu-id="4274a-175">This challenge is also related to the question of how to propagate changes across multiple microservices when certain data needs to be redundant—for example, when you need to have the product's name or description in the Catalog microservice and the Basket microservice.</span></span>

## <a name="challenge-4-how-to-design-communication-across-microservice-boundaries"></a><span data-ttu-id="4274a-178">Wyzwanie \#4: jak projektować komunikację między granicami mikrousług</span><span class="sxs-lookup"><span data-stu-id="4274a-178">Challenge \#4: How to design communication across microservice boundaries</span></span>

<span data-ttu-id="4274a-179">Komunikacja między granicami mikrousług jest rzeczywistym wyzwaniem.</span><span class="sxs-lookup"><span data-stu-id="4274a-179">Communicating across microservice boundaries is a real challenge.</span></span> <span data-ttu-id="4274a-180">W tym kontekście komunikacja nie odnosi się do używanego protokołu (HTTP i REST, AMQP, Messaging itd.).</span><span class="sxs-lookup"><span data-stu-id="4274a-180">In this context, communication doesn't refer to what protocol you should use (HTTP and REST, AMQP, messaging, and so on).</span></span> <span data-ttu-id="4274a-181">Zamiast tego określa styl komunikacji, który powinien być używany, a zwłaszcza na to, jak to mikrousługi.</span><span class="sxs-lookup"><span data-stu-id="4274a-181">Instead, it addresses what communication style you should use, and especially how coupled your microservices should be.</span></span> <span data-ttu-id="4274a-182">W zależności od poziomu sprzęgu, gdy wystąpi awaria, wpływ tego błędu na system różni się znacznie.</span><span class="sxs-lookup"><span data-stu-id="4274a-182">Depending on the level of coupling, when failure occurs, the impact of that failure on your system will vary significantly.</span></span>
