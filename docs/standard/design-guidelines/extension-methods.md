# <a name="extension-methods"></a><span data-ttu-id="1b481-102">Metody rozszerzania</span><span class="sxs-lookup"><span data-stu-id="1b481-102">Extension Methods</span></span>
<span data-ttu-id="1b481-103">Metody rozszerzające są funkcją języka, która umożliwia wywoływanie metod statycznych przy użyciu składni wywołania metody wystąpienia.</span><span class="sxs-lookup"><span data-stu-id="1b481-103">Extension methods are a language feature that allows static methods to be called using instance method call syntax.</span></span> <span data-ttu-id="1b481-104">Te metody muszą przyjmować co najmniej jeden parametr, który reprezentuje wystąpienie, na którym ma działać Metoda.</span><span class="sxs-lookup"><span data-stu-id="1b481-104">These methods must take at least one parameter, which represents the instance the method is to operate on.</span></span>

 <span data-ttu-id="1b481-105">Klasa, która definiuje takie metody rozszerzenia, jest określana jako Klasa "sponsora" i musi być zadeklarowana jako statyczna.</span><span class="sxs-lookup"><span data-stu-id="1b481-105">The class that defines such extension methods is referred to as the "sponsor" class, and it must be declared as static.</span></span> <span data-ttu-id="1b481-106">Aby można było użyć metod rozszerzających, należy zaimportować przestrzeń nazw definiującą klasę sponsora.</span><span class="sxs-lookup"><span data-stu-id="1b481-106">To use extension methods, one must import the namespace defining the sponsor class.</span></span>

 <span data-ttu-id="1b481-107">❌Należy unikać frivolously definiowania metod rozszerzających, szczególnie w przypadku typów, które nie są posiadane.</span><span class="sxs-lookup"><span data-stu-id="1b481-107">❌ AVOID frivolously defining extension methods, especially on types you don't own.</span></span>

 <span data-ttu-id="1b481-108">Jeśli używasz kodu źródłowego typu, rozważ użycie zwykłych metod wystąpienia.</span><span class="sxs-lookup"><span data-stu-id="1b481-108">If you do own source code of a type, consider using regular instance methods instead.</span></span> <span data-ttu-id="1b481-109">Jeśli nie jesteś własnością, i chcesz dodać metodę, należy zachować ostrożność.</span><span class="sxs-lookup"><span data-stu-id="1b481-109">If you don't own, and you want to add a method, be very careful.</span></span> <span data-ttu-id="1b481-110">Zliberalizowane korzystanie z metod rozszerzających ma możliwość bałaganu interfejsów API typów, które nie zostały zaprojektowane w taki sposób, aby miały te metody.</span><span class="sxs-lookup"><span data-stu-id="1b481-110">Liberal use of extension methods has the potential of cluttering APIs of types that were not designed to have these methods.</span></span>

 <span data-ttu-id="1b481-111">✔️ ROZWAŻYĆ użycie metod rozszerzających w jednym z następujących scenariuszy:</span><span class="sxs-lookup"><span data-stu-id="1b481-111">✔️ CONSIDER using extension methods in any of the following scenarios:</span></span>

- <span data-ttu-id="1b481-112">Aby zapewnić funkcje pomocnika odpowiednie dla każdej implementacji interfejsu, jeśli powyższe funkcje mogą być zapisywane w postaci podstawowego interfejsu.</span><span class="sxs-lookup"><span data-stu-id="1b481-112">To provide helper functionality relevant to every implementation of an interface, if said functionality can be written in terms of the core interface.</span></span> <span data-ttu-id="1b481-113">Wynika to z tego, że w przeciwnym razie nie można przypisać konkretnych implementacji do interfejsów.</span><span class="sxs-lookup"><span data-stu-id="1b481-113">This is because concrete implementations cannot otherwise be assigned to interfaces.</span></span> <span data-ttu-id="1b481-114">Na przykład `LINQ to Objects` Operatory są implementowane jako metody rozszerzające dla wszystkich <xref:System.Collections.Generic.IEnumerable%601> typów.</span><span class="sxs-lookup"><span data-stu-id="1b481-114">For example, the `LINQ to Objects` operators are implemented as extension methods for all <xref:System.Collections.Generic.IEnumerable%601> types.</span></span> <span data-ttu-id="1b481-115">W rezultacie każda `IEnumerable<>` implementacja jest automatycznie włączona w LINQ.</span><span class="sxs-lookup"><span data-stu-id="1b481-115">Thus, any `IEnumerable<>` implementation is automatically LINQ-enabled.</span></span>

- <span data-ttu-id="1b481-116">Gdy metoda wystąpienia wprowadzi zależność od pewnego typu, ale taka zależność spowoduje przerwanie reguł zarządzania zależnościami.</span><span class="sxs-lookup"><span data-stu-id="1b481-116">When an instance method would introduce a dependency on some type, but such a dependency would break dependency management rules.</span></span> <span data-ttu-id="1b481-117">Na przykład zależność od do elementu <xref:System.String> <xref:System.Uri?displayProperty=nameWithType> prawdopodobnie nie jest pożądana, a więc `String.ToUri()` metoda wystąpienia zwraca `System.Uri` niewłaściwy projekt z perspektywy zarządzania zależnościami.</span><span class="sxs-lookup"><span data-stu-id="1b481-117">For example, a dependency from <xref:System.String> to <xref:System.Uri?displayProperty=nameWithType> is probably not desirable, and so `String.ToUri()` instance method returning `System.Uri` would be the wrong design from a dependency management perspective.</span></span> <span data-ttu-id="1b481-118">Zwracana Metoda rozszerzenia statycznego będzie `Uri.ToUri(this string str)` `System.Uri` znacznie lepszym projektem.</span><span class="sxs-lookup"><span data-stu-id="1b481-118">A static extension method `Uri.ToUri(this string str)` returning `System.Uri` would be a much better design.</span></span>

 <span data-ttu-id="1b481-119">❌Należy unikać definiowania metod rozszerzających <xref:System.Object?displayProperty=nameWithType> .</span><span class="sxs-lookup"><span data-stu-id="1b481-119">❌ AVOID defining extension methods on <xref:System.Object?displayProperty=nameWithType>.</span></span>

 <span data-ttu-id="1b481-120">Użytkownicy VB nie będą mogli wywoływać takich metod w odwołaniach do obiektów przy użyciu składni metody rozszerzenia.</span><span class="sxs-lookup"><span data-stu-id="1b481-120">VB users will not be able to call such methods on object references using the extension method syntax.</span></span> <span data-ttu-id="1b481-121">VB nie obsługuje wywoływania takich metod, ponieważ w języku VB deklarując odwołanie jako obiekt wymusza, że wszystkie wywołania metody na nim mają być późne (rzeczywista nazwa elementu członkowskiego jest określana w czasie wykonywania), podczas gdy powiązania z metodami rozszerzania są określane w czasie kompilacji (wczesnym wiązaniem).</span><span class="sxs-lookup"><span data-stu-id="1b481-121">VB does not support calling such methods because, in VB, declaring a reference as Object forces all method invocations on it to be late bound (actual member called is determined at runtime), while bindings to extension methods are determined at compile-time (early bound).</span></span>

 <span data-ttu-id="1b481-122">Należy zauważyć, że wytyczne dotyczą innych języków, w których występuje takie samo zachowanie dotyczące powiązań lub których metody rozszerzające nie są obsługiwane.</span><span class="sxs-lookup"><span data-stu-id="1b481-122">Note that the guideline applies to other languages where the same binding behavior is present, or where extension methods are not supported.</span></span>

 <span data-ttu-id="1b481-123">❌NIE należy umieszczać metod rozszerzających w tej samej przestrzeni nazw co typ rozszerzony, chyba że służy do dodawania metod do interfejsów ani do zarządzania zależnościami.</span><span class="sxs-lookup"><span data-stu-id="1b481-123">❌ DO NOT put extension methods in the same namespace as the extended type unless it is for adding methods to interfaces or for dependency management.</span></span>

 <span data-ttu-id="1b481-124">❌Należy unikać definiowania dwóch lub więcej metod rozszerzających o tej samej sygnaturze, nawet jeśli znajdują się one w różnych przestrzeniach nazw.</span><span class="sxs-lookup"><span data-stu-id="1b481-124">❌ AVOID defining two or more extension methods with the same signature, even if they reside in different namespaces.</span></span>

 <span data-ttu-id="1b481-125">✔️ ROZWAŻYĆ zdefiniowanie metod rozszerzenia w tej samej przestrzeni nazw, co typ rozszerzony, jeśli typ jest interfejsem i jeśli metody rozszerzające mają być używane w większości lub we wszystkich przypadkach.</span><span class="sxs-lookup"><span data-stu-id="1b481-125">✔️ CONSIDER defining extension methods in the same namespace as the extended type if the type is an interface and if the extension methods are meant to be used in most or all cases.</span></span>

 <span data-ttu-id="1b481-126">❌NIE należy definiować metod rozszerzających implementujących funkcję w przestrzeniach nazw zwykle skojarzonych z innymi funkcjami.</span><span class="sxs-lookup"><span data-stu-id="1b481-126">❌ DO NOT define extension methods implementing a feature in namespaces normally associated with other features.</span></span> <span data-ttu-id="1b481-127">Zamiast tego należy je zdefiniować w przestrzeni nazw skojarzonej z funkcją, do której należą.</span><span class="sxs-lookup"><span data-stu-id="1b481-127">Instead, define them in the namespace associated with the feature they belong to.</span></span>

 <span data-ttu-id="1b481-128">❌UNIKAj ogólnego nazewnictwa przestrzeni nazw przeznaczonych dla metod rozszerzających (np. "rozszerzeń").</span><span class="sxs-lookup"><span data-stu-id="1b481-128">❌ AVOID generic naming of namespaces dedicated to extension methods (e.g., "Extensions").</span></span> <span data-ttu-id="1b481-129">Zamiast tego użyj nazwy opisowej (np. "routingu").</span><span class="sxs-lookup"><span data-stu-id="1b481-129">Use a descriptive name (e.g., "Routing") instead.</span></span>
